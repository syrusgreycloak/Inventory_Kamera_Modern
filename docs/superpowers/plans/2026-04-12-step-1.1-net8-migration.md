# Step 1.1: .NET 8 Migration Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Migrate InventoryKamera from .NET Framework 4.7.2 to .NET 8 (net8.0-windows), keeping the app fully functional — same scan, same GOOD JSON output.

**Architecture:** Replace the legacy-format `.csproj` with SDK-style format targeting `net8.0-windows`. Remove .NET Framework-only packages that .NET 8 provides natively. Replace the one hard blocker (`Thread.Abort()` at line 92 in MainForm.cs) with cooperative cancellation via `CancellationTokenSource`. Everything else compiles as-is.

**Tech Stack:** .NET 8 SDK, MSBuild (Rider's), WinForms (`UseWindowsForms`), NuGet PackageReference. No new runtime dependencies.

**What was removed and why:**
- `HtmlAgilityPack.NetCore` — check whether it's actually used: `grep -r "HtmlAgilityPack" InventoryKamera/` — if no results, it was an unused transitive dep. Drop it.
- `Microsoft.Win32.Primitives`, `System.Collections`, `System.Diagnostics.DiagnosticSource`, `System.Runtime`, `System.Security.Cryptography.*` — these are .NET Framework polyfill packages. .NET 8 provides them natively; including them causes version conflicts.
- `NHotkey.WindowsForms` bumped from 2.1.0 → 3.0.0 (3.x targets .NET 6+; 2.x targets .NET Framework).
- `Newtonsoft.Json` bumped 13.0.2 → 13.0.3 (patch release, same API).
- `NLog` bumped 5.1.1 → 5.3.4 (same major, .NET 8 compatible).
- `Octokit` bumped 7.1.0 → 13.0.1 (7.x is .NET Framework only).
- The `<Reference Include="...">` framework assembly list is dropped — SDK-style WinForms projects include them automatically via `UseWindowsForms`.
- All `<Compile Include="...">` entries dropped — SDK-style auto-includes all `.cs` files.
- `<EmbeddedResource>` entries for `.resx` files dropped — auto-discovered by SDK.

## Task 3: Fix Thread.Abort() — replace with CancellationTokenSource

`Thread.Abort()` throws `PlatformNotSupportedException` at runtime on .NET 8 (it compiles but crashes immediately when called). `ThreadAbortException` is also removed from .NET 8, so the `catch (ThreadAbortException)` block at MainForm.cs:284 won't compile.

The fix: add a `CancellationTokenSource` to `MainForm`. When the hotkey fires, cancel the source instead of aborting the thread. The existing `b_threadCancel` flag in `InventoryKamera.cs:32` already makes the scan loops exit cooperatively — wire the cancellation through to it.

**Files:**
- Modify: `InventoryKamera/ui/main/MainForm.cs`

- [ ] **Step 1: Add the CancellationTokenSource field**

In `MainForm.cs`, the field declarations start at line 26. Add `_cts` alongside the existing `scannerThread` field:

```csharp
// line 27 — existing:
private static Thread scannerThread;
// add after it:
private CancellationTokenSource _cts;
```

- [ ] **Step 2: Initialize _cts when the scan starts**

In `StartButton_Clicked`, the scan thread is created at line 242. Initialize `_cts` just before creating the thread:

```csharp
// Add immediately before: scannerThread = new Thread(() =>
_cts = new CancellationTokenSource();
scannerThread = new Thread(() =>
{
    // ... existing body unchanged for now ...
```

- [ ] **Step 3: Replace Thread.Abort() in Hotkey_Pressed**

The current code at lines 89–97:
```csharp
if (scannerThread.IsAlive)
{
    // Stop navigating weapons/artifacts
    scannerThread.Abort();

    UserInterface.SetProgramStatus("Scan Stopped");

    Navigation.Reset();
}
```

Replace with:
```csharp
if (scannerThread != null && scannerThread.IsAlive)
{
    // Signal cooperative cancellation — scan loops check b_threadCancel
    _cts?.Cancel();
    data?.StopImageProcessorWorkers();

    UserInterface.SetProgramStatus("Scan Stopped");

    Navigation.Reset();
}
```

- [ ] **Step 4: Replace the ThreadAbortException catch block**

The current catch block at lines 284–289:
```csharp
catch (ThreadAbortException)
{
    // Workers can get stuck if the thread is aborted or an exception is raised
    data?.StopImageProcessorWorkers();
    UserInterface.SetProgramStatus("Scan stopped");
}
```

Replace with:
```csharp
catch (OperationCanceledException)
{
    // Hotkey pressed — cooperative cancellation
    data?.StopImageProcessorWorkers();
    UserInterface.SetProgramStatus("Scan stopped");
}
```

## Troubleshooting: Common .NET 8 migration errors

**`CS0246: The type or namespace name 'ThreadAbortException' could not be found`**
→ Fixed in Task 3 Step 4. Replace with `OperationCanceledException`.

**`InputSimulator` namespace not found or API mismatch**
→ `InputSimulator 1.0.4` targets .NET Framework. Replace in csproj with:
```xml
<PackageReference Include="InputSimulatorPlus" Version="1.0.8" />
```
The API is identical — `new InputSimulator()`, `Keyboard.KeyPress()`, etc.

**`NHotkey.WindowsForms` not found**
→ Ensure version 3.0.0 is specified. Version 2.x targets .NET Framework only.

**`Accord.Imaging` compile errors**
→ Note them but don't fix them here — Accord.Imaging is scheduled for removal in Step 1.2 (Core library extraction). If errors are blocking the build, temporarily comment out the Accord.Imaging `using` directives and the specific call sites.

**`Microsoft-WindowsAPICodePack-Shell` not found**
→ This package is used for the folder picker dialog (`CommonOpenFileDialog` in MainForm.cs). Try `Microsoft-WindowsAPICodePack-Shell 1.1.5` first. If NuGet can't resolve it, replace the folder picker with the standard `FolderBrowserDialog`:
```csharp
// Replace CommonOpenFileDialog usage with:
using var d = new FolderBrowserDialog
{
    InitialDirectory = !Directory.Exists(OutputPath_TextBox.Text)
        ? Directory.GetCurrentDirectory()
        : OutputPath_TextBox.Text
};
if (d.ShowDialog() == DialogResult.OK)
{
    OutputPath_TextBox.Text = d.SelectedPath;
}
```

**Settings not persisting between runs**
→ .NET 8 user settings are stored in a different path than .NET Framework. This is expected — users will need to reconfigure settings once after upgrading. Not a bug.
